# Goalrail Short Form Content System v0

> Рабочая система для short-form content вокруг Goalrail / Goal on Rails / Punk.
> Это первый operational brand doc после character system.
> Его задача — зафиксировать content grammar раньше, чем финальную visual identity.

## 1. Purpose

Этот документ фиксирует:
- зачем short-form content нужен проекту сейчас
- почему он идёт раньше visual identity
- какие типы роликов и каруселей делаем по умолчанию
- какой narrative grammar используется
- как использовать Punk в short-form
- как оформлять subtitles / captions / CTA

## 2. Why this doc comes before Visual Identity

Current recommendation:

**сначала Short Form Content System, потом Visual Identity v0.**

Причина:
- narrative уже формируется
- mascot role уже зафиксирован
- product story уже начинает собираться
- а visual system ещё естественно будет эволюционировать

Сначала нужно стабилизировать:
- что говорим
- как говорим
- в каком порядке подаём мысль
- как выглядит один рабочий контент-юнит

И только потом имеет смысл более жёстко фиксировать:
- palette
- typography
- graphic motifs
- asset kit

## 3. Primary objective

Short-form content сейчас нужен прежде всего для:

**сборки business-side audience вокруг проблемы AI delivery drift и вокруг build-in-public истории Goalrail.**

Не primary objective:
- развлекать developer meme culture
- строить mascot fandom отдельно от продукта
- гнать pure visual experimentation без product narrative

## 4. Audience default

Оптимизация по умолчанию под:
- PM
- product lead
- tech lead
- CTO / Head of Engineering
- business-side sponsor

Secondary audience:
- developers
- AI builders
- operators

Rule:
short-form content остаётся **business-readable**, даже если визуально он sharp / cyberpunk.

## 5. Format default

Primary formats now:
- short vertical video
- talking mascot / voiced explainers
- carousel / comic-style cards
- simple narrated build updates

Default runtime target:
- 20–45 seconds for video
- 4–7 cards for carousel

## 6. Core content grammar

Default grammar:

**Pain -> Rail -> Result -> Proof -> CTA**

### Pain
Показываем, где AI delivery расползается.

### Rail
Показываем, как Goalrail ставит работу в bounded contour.

### Result
Показываем, какой управляемый outcome появляется.

### Proof
Подчёркиваем, что важен не просто patch / vibe / status,
а проверяемый результат.

### CTA
Даем следующий business-readable шаг.

## 7. Default content pillars

### 7.1 Why delivery drifts
Почему AI ускоряет execution, но не даёт управляемости сам по себе.

### 7.2 Put goals on rails
Что значит поставить goal на rails practically.

### 7.3 Building Goalrail in public
Что именно сейчас строится и почему.

### 7.4 Honest pilot motion
Как это должно заходить в компанию без platform theater.

### 7.5 Proof over vibes
Почему patch / PR / demo ещё не enough.

## 8. Default episode types

### 8.1 Problem clip
Один clip = одна боль.

Example:
- AI пишет код быстрее, чем команды успевают договориться о смысле.

### 8.2 Translation clip
Один clip = перевод сложной delivery idea на business language.

Example:
- что такое working contract простыми словами

### 8.3 Build log clip
Один clip = один реальный build step.

Example:
- сегодня зафиксировали public narrative layer

### 8.4 Punk signal clip
Один clip = один жёсткий принцип от Punk.

Example:
- Patch is not proof.

### 8.5 Pilot clip
Один clip = один честный тезис про то, как запускать pilot.

## 9. One-unit rule

Каждый short-form unit должен отвечать на **одну** из задач:
- объяснить одну проблему
- объяснить одну идею
- показать один build step
- зафиксировать один principle
- дать один CTA

Rule:
один контент-юнит не должен пытаться одновременно:
- объяснить весь продукт
- показать весь roadmap
- раскрыть весь lore
- продать весь pilot motion

## 10. Punk usage in short-form

Punk в short-form выполняет одну из ролей:
- signal
- guide
- critic
- builder

Rule:
Punk усиливает clip,
но не должен забирать весь смысл на себя.

By default:
- Punk opens the thought
- product or pain carries the substance
- CTA closes the unit

## 11. Video structure default

Для vertical video default structure такая:

1. **Hook** — 1 мысль / 1 ударная фраза
2. **Problem framing** — что идёт не так
3. **Rail / control idea** — что меняется
4. **Proof/result** — зачем это важно
5. **CTA** — что сделать дальше

## 12. Carousel structure default

Для carousel default structure такая:

1. Hook card
2. Problem card
3. Why this matters card
4. Rail / system answer card
5. Proof / result card
6. CTA card

Optional:
- card with Punk quote
- card with current build artifact

## 13. Hook rules

Хороший hook:
- короткий
- понятный бизнесу
- напряжённый
- problem-led

Examples:
- AI ускорил код. Не delivery.
- Goal without rails becomes drift.
- Patch — это ещё не результат.
- Самая дорогая ошибка — начать писать до согласования границ.

## 14. Caption rules

### 14.1 Caption purpose
Caption не должен просто дублировать spoken line.
Он должен:
- дать framing
- усилить мысль
- дать quick read value
- завершить CTA

### 14.2 Caption length
Default caption style:
- коротко
- 1 главный тезис
- 1 короткое раскрытие
- 1 CTA

### 14.3 Caption structure
Default structure:
- line 1: hook
- line 2–3: explanation
- last line: CTA

Example shape:
- AI execution is getting faster.
- Controlled delivery still isn’t solved.
- We’re building Goalrail in public.
- Send a task for pilot review.

## 15. Subtitle rules

### 15.1 Subtitle purpose
Subtitles нужны не только для accessibility,
но и как часть comprehension layer.

### 15.2 Subtitle style
Default subtitle style should be:
- short chunks
- one thought per line
- no dense paragraphs
- easy to scan without audio

### 15.3 Reading rule
Если фраза не читается быстро глазами,
она слишком длинная для subtitle line.

### 15.4 Emphasis rule
Допустимо визуально подчеркивать только load-bearing words:
- goal
- rails
- drift
- proof
- pilot
- verify

Не нужно выделять всё подряд.

### 15.5 Sync rule
Subtitles должны быть чуть более readable,
чем raw transcript.

То есть допустимо:
- резать длинные фразы
- убирать мусорные связки
- делать мысль чище

Но нельзя silently менять смысл.

### 15.6 Tone rule
Даже в subtitles сохраняется same tone:
- concise
- sharp
- business-readable

## 16. CTA defaults

Preferred CTA set:
- Получить пилотный разбор
- Прислать задачу на разбор
- Посмотреть walkthrough
- Follow the build in public

Avoid CTA set:
- buy now
- replace your team
- automate everything
- launch full autopilot

## 17. Evidence rule

Когда возможно, short-form content должен ссылаться на real build evidence:
- doc update
- design exploration
- flow diagram
- repo decision
- pilot framing
- new layer in documentation

Это особенно важно для build log clips.

## 18. Production rule v0

Current v0 production principle:
- ship simple
- keep consistent structure
- improve later

Это значит:
- не ждать идеальной visual identity
- не ждать идеального mascot rig
- не ждать киношного монтажа
- сначала стабилизировать content grammar

## 19. Relationship to future docs

Этот документ должен стать входом перед:
- `VISUAL_IDENTITY_V0.md`
- future asset rules
- future subtitle / motion refinements

Visual Identity v0 later should answer:
- how it looks

This doc answers first:
- what one content unit does
- how it is structured
- how it speaks

## 20. Non-goals

Этот документ пока не фиксирует:
- exact palette
- exact typography system
- exact thumbnail system
- exact editing workflow
- platform-specific posting calendar
- final motion language

## 21. Current recommendation summary

**Short Form Content System comes first because the project needs a stable content grammar before it needs a stable visual identity. Subtitle and caption rules are part of that grammar and should be fixed now at v0 level, not postponed.**
