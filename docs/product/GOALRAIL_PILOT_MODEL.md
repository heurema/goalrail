# Goalrail Pilot Model

## 1. Purpose

Этот документ описывает первый честный коммерческий формат Goalrail.

Pilot нужен не для того, чтобы “пообещать платформу в будущем”.
Pilot нужен, чтобы на реальном кейсе показать:

- работает ли productized operating model
- уменьшает ли он ambiguity
- даёт ли он bounded AI-assisted delivery
- создаёт ли он inspectable proof

## 2. Pilot principle

Pilot — это не широкий rollout.
Pilot — это не общий аудит всей компании.
Pilot — это не бесконечный консалтинг.

Pilot — это:

- one team
- one or two repos
- one or two real cases
- one visible flow from task to proof

## 3. Recommended pilot shape

### Default recommendation
- 1 команда
- 1 repo на старте
- 1 основной кейс
- 2 недели как базовый срок
- managed rollout motion
- lightweight proof required from day one

### Extended recommendation
- 1 команда
- 1–2 репо
- 2 кейса
- 3–4 недели
- когда клиенту нужен более уверенный expansion decision

## 4. Pilot phases

### Phase A — Qualification
Проверяем fit и выбираем pilot case.

### Phase B — Bootstrap
Подключаем repo, runtime, templates, policy defaults.

### Phase C — Onboarding
Команда понимает новый operating model.

### Phase D — Pilot run
Реальная задача проходит через flow:
task -> contract -> execution -> verify -> proof

### Phase E — Readout
Показываем результат и принимаем expansion decision.

## 5. What goes into pilot

В пилот входит:

- qualification / fit check
- profile selection
- one deployment configuration
- initial bootstrap
- one onboarding session
- one real pilot case
- one proof output
- one readout with rollout recommendation

## 6. What does not go into pilot

В пилот не входит:

- полная замена toolchain
- full enterprise governance rollout
- бесконечная кастомизация под оргспецифику клиента
- автоматизация всех delivery workflows
- bespoke redesign всей команды

## 7. Pilot inputs

Клиент должен дать:

- sponsor
- one team
- one real use case
- one repo or clear task input path
- access necessary for bounded deployment
- readiness to run new workflow on a live case

## 8. Pilot outputs

На выходе pilot клиент получает:

- configured Goalrail operating profile
- one real contract-driven delivery case
- one proof artifact
- readout of what worked / what blocked
- rollout recommendation

## 9. Pilot success criteria

Pilot успешен, если:

1. реальная задача была проведена через Goalrail flow
2. contract оказался реально полезным working object
3. execution был bounded и reviewable
4. результат завершился inspectable proof
5. команда хочет расширять применение или осознанно говорит no-fit

## 10. Pricing posture recommendation

Рекомендуемый коммерческий posture:

- **free diagnostic / qualification**
- **paid pilot**

Почему:
- qualification нужен, чтобы быстро убрать no-fit cases
- основной value создаётся на pilot execution, а не на разговоре
- paid pilot лучше валидирует seriousness клиента

## 11. Remote vs onsite recommendation

Рекомендуемый default:
- **remote-first**

Onsite — только по необходимости:
- security-sensitive context
- executive alignment workshop
- later expansion phases

## 12. Commercial CTA

Первый CTA должен быть не “купить подписку”.

Правильные CTA:
- прислать задачу
- открыть pilot-разбор
- пройти qualification
- запустить pilot

## 13. End state

Pilot не обязан завершаться full rollout.
Он обязан завершаться ясным решением:

- expand
- stabilize and retry
- stop
