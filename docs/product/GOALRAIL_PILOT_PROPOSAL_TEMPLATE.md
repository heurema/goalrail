# Goalrail Pilot Proposal Template

> Draft template.
> Использовать как рабочую основу для первых pilot proposals.
> Не считать финальным commercial format до стабилизации offer и первых closed-loop pilots.

## Purpose

Этот документ нужен как шаблон короткого pilot proposal после qualification call.

Его задача:
- быстро перевести интерес клиента в bounded commercial proposal
- не скатиться в vague consulting scope
- держать один ясный объект продажи: **managed pilot**

## How to use

1. Скопировать шаблон.
2. Заполнить только конкретику по клиенту.
3. Не добавлять широкие обещания вне pilot scope.
4. Если нужен больший объём — не раздувать этот документ, а явно re-scope the pilot.

## 2. Executive summary

Goalrail предлагается как **pilot-first operating layer for AI-assisted delivery**.

Цель данного pilot:
- провести **один реальный кейс** через управляемый flow
- превратить входящую задачу в bounded execution path
- вернуть **inspectable proof**, а не просто статус

В рамках этого proposal мы предлагаем:
- 1 команду
- 1 repo
- 1 основной кейс
- 2-недельный managed pilot

## 3. Why this pilot

Текущая проблема, которую pilot должен проверить:
- [brief ambiguity / speed / reviewability / AI adoption pain]

Мы хотим проверить:
- помогает ли contract-first flow уменьшить ambiguity
- можно ли провести bounded AI-assisted delivery без потери контроля
- может ли команда получить inspectable proof по реальному кейсу

## 4. Pilot scope

### Included
- qualification confirmation
- pilot profile selection
- bootstrap for 1 repo
- 1 onboarding session
- 1 real pilot case
- bounded execution flow
- 1 proof output
- final readout with rollout recommendation

### Not included
- full platform rollout
- replacement of existing trackers / IDE / toolchain
- open-ended consulting
- full redesign of client engineering process
- broad multi-team deployment

## 6. Delivery model

### Working model
This pilot is delivered as a **founder-led managed pilot**.

### Delivery cell
- **Founder** — qualification, pilot framing, contract logic, readout
- **DevOps** — bootstrap, runtime/setup, environment and delivery constraints where needed

### Engagement format
- remote-first
- async working channel: [Telegram / Slack / email]
- check-ins: [2x per week or similar]
- one final readout session

## 7. Pilot phases and timeline

### Week 0 / Pre-start
- confirm case
- confirm repo and access
- confirm sponsor and working contacts
- lock pilot window

### Week 1
- profile selection
- bootstrap
- onboarding
- contract shaping for pilot case
- first bounded execution pass

### Week 2
- execution continuation / stabilization
- verification and proof packaging
- final readout
- expansion / retry / stop recommendation

## 8. Client inputs required

Client is expected to provide:
- one sponsor
- one real pilot case
- access to one repo or clear task input path
- team availability for onboarding and review
- realistic success criteria for the case
- security and environment constraints upfront

## 9. Deliverables

At pilot close the client receives:
- configured pilot profile
- one real contract-driven case walkthrough
- one proof artifact
- readout of what worked / what blocked
- explicit recommendation:
  - expand
  - stabilize and retry
  - stop

## 10. Success criteria

This pilot is considered successful if:
1. one real task passes through the Goalrail flow
2. the contract becomes a useful working object
3. execution remains bounded and reviewable
4. the result ends with inspectable proof
5. the team can make a grounded rollout decision

## 11. Commercial terms

### Pilot package
**Managed pilot — from $5,000**

### This proposal
**Quoted price:** [$X,XXX]

### Payment terms
- 60% upfront
- 40% on final readout / pilot close

### Notes
- For early design-partner accounts, a discounted pilot may be offered explicitly.
- Additional repos, additional cases, or heavy policy constraints are re-scoped separately.

## 12. Assumptions and exclusions

### Assumptions
- client can provide required access on time
- one real case is available during the pilot window
- the team is willing to use a bounded pilot flow

### Exclusions
- no guaranteed full rollout after pilot
- no promise of org-wide transformation inside this engagement
- no claim that Goalrail is already a finished platform product

## 13. Next step

If the client wants to proceed:
1. confirm pilot case
2. confirm participants and access
3. confirm pilot window
4. approve proposal and commercial terms
5. schedule kickoff
