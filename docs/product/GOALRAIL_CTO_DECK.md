# Goalrail CTO Deck

> Canonical outline for the first Goalrail sales deck.
>
> Audience:
> - CTO
> - Head of Engineering
> - VP Engineering
>
> Goal:
> explain the problem, the operating model, the pilot format, and the next commercial step without turning the conversation into a product fantasy or architecture deep dive.

## 1. Deck principle

This deck is not:
- a startup fundraising deck
- a long product spec
- a technical architecture review
- an “AI magic” presentation

This deck is:
- a decision-support artifact for a pilot conversation
- a way to show what Goalrail changes in the delivery loop
- a path from problem -> operating model -> pilot -> expansion decision

## 2. Recommended deck length

Recommended length:
- **6–8 slides**

Why:
- enough to explain the problem and the pilot
- short enough for CTO attention span
- leaves room for conversation and questions

## 3. Slide-by-slide structure

### Slide 1 — Title / one-line product thesis
**Goalrail — от бизнес-цели до проверенного изменения в коде**

Supporting line:
Goalrail is a productized operating layer for AI-assisted delivery.

Speaker note:
Do not lead with agents or models.
Lead with the delivery-control problem.

---

### Slide 2 — The problem
Title:
**AI accelerates execution, but does not solve the gap between task and trustworthy result**

Core points:
- incoming tasks are often vague
- scope and constraints are weakly captured
- AI starts writing before the team agrees on boundaries
- output ends as patch / diff / PR without shared proof
- leadership sees activity, but not always inspectable trust

Speaker note:
This slide should sound like a known operational pain, not a visionary category invention.

---

### Slide 3 — What Goalrail changes
Title:
**Goalrail adds one controlled layer between incoming task and engineering outcome**

Show the flow:
`incoming task -> working contract -> bounded execution -> verify -> proof`

Core message:
Goalrail does not replace the current stack.
It adds a shared working contract, controlled execution boundaries, and proof-oriented visibility.

---

### Slide 4 — The operating model
Title:
**Contract first. Execution second. Proof required.**

Core points:
- task becomes a working contract before execution
- execution is bounded
- one writable run has one primary writer runtime
- final evaluation is separate from execution
- output is proof, not just status

Speaker note:
Keep this operational, not technical.
This is the heart of the story.

---

### Slide 5 — How deployment works
Title:
**Goalrail starts as a productized pilot, not a broad transformation project**

Show the rollout path:
1. qualification
2. profile selection
3. bootstrap
4. onboarding
5. pilot run
6. proof / readout
7. expansion decision

Core message:
We do not ask for a full-stack commitment first.
We prove value on one bounded real case.

---

### Slide 6 — What the pilot includes
Title:
**First honest commercial format: one team, one repo, one visible loop**

Suggested bullets:
- 1 team
- 1 repo on start
- 1 real case
- 2 weeks baseline
- managed deployment
- proof from day one

Outputs:
- configured pilot profile
- one contract-driven case
- one proof artifact
- rollout recommendation

---

### Slide 7 — Where Goalrail fits
Title:
**Not another AI IDE. Not a replacement for Jira. Not a generic agent framework.**

Left column: what Goalrail is not
- not another coding assistant
- not a tracker replacement
- not a giant all-in-one platform
- not bespoke consulting per company

Right column: what Goalrail is
- productized operating layer
- shared source of truth for AI-assisted delivery
- bounded execution + verify / proof contour
- supplement layer over existing tools

---

### Slide 8 — Next step / CTA
Title:
**Start with one task and one pilot decision**

CTA options:
- Send one real task for review
- Run qualification
- Start a paid pilot

Closing message:
We do not need a full rollout decision first.
We need one real case, one bounded deployment, and one proof-based decision.

## 4. Visual recommendations

Recommended tone:
- dark / restrained / premium
- editorial, not startup-hype
- minimal diagrams
- one strong flow per slide
- low interface noise

Recommended visual anchor:
- working contract as the central object
- simple flow from task to proof

Avoid:
- prompt screenshots as the hero
- agent buttons as the hero
- dashboard-heavy clutter
- feature matrix overload

## 5. Messaging rules for the deck

Lead with:
- ambiguity between task and result
- controlled AI-assisted delivery
- shared source of truth
- bounded execution
- proof-oriented visibility

Do not lead with:
- models
- orchestration jargon
- MCP
- benchmarks
- provider comparisons
- architecture internals

## 6. Suggested speaking rhythm

Suggested order when presenting:
1. state the operational problem
2. show the new contour
3. explain how the pilot works
4. explain what the client gets
5. ask for one real case
