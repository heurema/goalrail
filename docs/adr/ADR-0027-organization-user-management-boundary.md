## Context

ADR-0022 defines `Installation` above `Organization`, and ADR-0023 defines
self-hosted bootstrap, auth credentials, token direction, and
browser-loopback CLI login.

The first self-hosted owner is created by `goalrail-server bootstrap owner`.
Public registration is out of MVP. Later users are expected to be created
inside the Organization by an existing administrator, receive a temporary
password, and change that password on first login.

The current repository already has auth credential/session foundations, the
bootstrap owner command, login/password-change endpoints, CLI login, and a
Console Settings / Users component-state UI. It does not have backend admin
user-management routes or persisted Console Users behavior.

Before adding those implementation slices, Goalrail needs one explicit boundary
for where Organization user management lives, which identity objects are
canonical, how authorization is checked, and what the CLI must not become.

This ADR is documentation-only. It defines direction for future product/admin
routes and Console persistence work. It does not implement backend code,
frontend code, migrations, routes, services, handlers, stores, or UI behavior.

## Decision

Regular Organization users are created through the Goalrail Console UI backed
by the server API.

Regular Organization users are not created through the Goalrail CLI in v0.
There is no separate CLI-user entity.

The canonical identity object is `User`.

Organization access is represented by `OrganizationMembership`.

Password credentials remain separate from `users`, continuing the ADR-0023
direction for `user_password_credentials` or an equivalent credential store.

Temporary passwords are generated by the backend, shown once to the authorized
admin at creation time, and require first-login password change before normal
product use.

The v0 Organization-management role vocabulary is:

- `owner`
- `admin`
- `member`
- `viewer`

Console roles must align with backend `OrganizationMembership` roles. The
documented target does not keep a separate `observer` role.

## Authorization

For v0, user management is owner-only.

The server must load the current caller `OrganizationMembership` server-side
for the target `organization_id` before allowing user-management reads or
writes.

JWTs must not be trusted for broad or stale role state. JWT access tokens may
authenticate the caller identity/session, but authorization must use current
server-side membership state.

The server must reject cross-organization user-management attempts. A caller
authorized in one Organization may not list, create, update, disable, or change
membership roles for users in another Organization.

The server must prevent disabling or demoting the last active owner in an
Organization.

## API direction

Future product/admin routes should be Organization-scoped:

- `GET /v1/organizations/{organization_id}/users`
- `POST /v1/organizations/{organization_id}/users`
- `PATCH /v1/organizations/{organization_id}/users/{user_id}`

These routes are future product/admin routes only. This ADR does not implement
or register them.

The list route should return Organization user and membership summary data. It
must not return password hashes, temporary passwords, refresh tokens, session
tokens, CLI auth codes, or credential material.

The create route should create a `User`, create or attach an
`OrganizationMembership`, create a password credential with
`must_change_password = true`, and return the backend-generated temporary
password exactly once in the create response.

The patch route should update bounded user-management fields such as active
state and Organization role. It must enforce the authorization rules in this
ADR, including cross-organization rejection and last-active-owner protection.

The API must not accept a plaintext password supplied by the Console for normal
admin-created users in v0. The backend owns temporary password generation.

## Password and session direction

Temporary passwords must never be persisted in plaintext.

Temporary passwords must never be logged.

Temporary passwords must never be returned from list or get endpoints.

Temporary passwords must never be stored in browser storage, including
`localStorage`, `sessionStorage`, IndexedDB, or cookies.

Only the password hash or equivalent credential verifier belongs in the server
credential store.

First-login password change remains mandatory for temporary-password users.
Users with `must_change_password = true` must not receive normal product or CLI
runtime access until they complete password change.

Session and token behavior remains aligned with ADR-0023: short-lived JWT
access tokens for narrow identity/session claims, opaque DB-backed refresh
token state, and server-side membership checks for authorization.

## Console direction

Settings / Users is the intended v0 product surface for Organization user
management.

The Console should call the future Organization-scoped server API for listing,
creating, disabling/enabling, and role changes. It should not invent local-only
canonical user state once the API exists.

The Console may display a generated temporary password immediately after a
successful create response, but must treat it as a one-time secret. It must not
persist that value in browser storage or reloadable state.

Console copy and role controls must use backend-aligned roles:

- `owner`
- `admin`
- `member`
- `viewer`

The Console should not present `observer` as a separate target role.

## CLI boundary

The CLI is for login and delivery/runtime commands after a user already exists.

`goalrail login <server_url>` remains the CLI authentication path for
already-created users.

The CLI must not introduce `goalrail users create` in v0.

The CLI must not become an Organization user-management surface, a separate
identity system, or the canonical owner of user state.

## Consequences

The next backend implementation slice can add Organization-scoped
user-management API behavior without reopening public registration, SaaS
organization creation, or CLI user creation.

The next Console implementation slice can replace component-state Users data
with API-backed persistence while preserving the existing no-token-storage
browser posture.

Authorization must stay organization-aware even in self-hosted mode with one
bootstrapped primary Organization.

Temporary-password handling becomes a one-time display and credential-hash
boundary, not a reusable secret transport or browser-persisted value.

The CLI remains compatible with self-hosted and future SaaS auth because it
logs in existing users instead of creating users.

## Rejected alternatives

### `goalrail users create` in v0

Rejected. User creation is an Organization administration workflow and belongs
in the Console backed by the server API. The CLI remains a login and
delivery/runtime adapter after identity already exists.

### Separate CLI-user entity

Rejected. The canonical identity is `User`; CLI access is a session/profile
view over an existing user, not a second identity model.

### Public registration

Rejected. Public registration would add SaaS onboarding, abuse handling,
organization creation, and invite/reset surface beyond the v0 boundary.

### Email invite or password reset delivery in v0

Rejected. Temporary-password handoff plus mandatory first-login password change
is the v0 direction. Invite and reset email delivery require a later bounded
slice.

### Trust JWT role claims for user management

Rejected. Role state can change. User management must use current server-side
`OrganizationMembership` state.

### Console-only local Users state as product truth

Rejected. The current component-state UI is temporary. Product user management
truth belongs to the server API and canonical persistence.

### Separate `observer` role

Rejected. The documented target role vocabulary is `owner`, `admin`, `member`,
and `viewer`.

## Non-goals

This ADR does not implement or define:

- backend API implementation
- frontend persistence implementation
- migrations
- routes
- services
- handlers
- stores
- UI behavior
- public registration
- invite email delivery
- password reset email delivery
- SaaS organization creation
- billing
- SSO/OIDC
- generic RBAC platform
- `goalrail users create`
- runner, gate, or proof behavior
